## 1. 我们在做什么

这是一个巴迪高品牌商业视觉生成工作台。当前阶段重点在重构“创意模式 / 替换模式”的状态与提示词链路，尤其是替换模式下的素材职责隔离、人物身份替换、多人绑定和原图单品替换准确率。

用户希望替换模式分成三条明确、互不串数据的工作流：

1. **姿势锁定重构**：保留动作参考图中的人物姿势，替换人物身份、产品和场景。
2. **原图单品替换**：原人物、构图、场景和光影不变，只替换目标产品。
3. **多要素精确替换**：分别控制人物形象、上身穿搭、下身穿搭、产品、动作、构图和场景。

总体目标是让每张上传图片只有一个清晰职责，避免动作图携带场景、构图图携带人物、人物图覆盖产品，或不同工作流之间复用上一轮数据。

- 顶部提供“创意模式 / 替换模式”切换。

- 切换工作流时会立即清空旧提示词预览，等待新提示词返回，避免短暂显示上一轮提示词。

主要实现位置：

工作流字段已进入项目状态、任务快照、前端提示词预览、提示词优化和真实生图请求。

### 2.4 多人模式

- 人数选择会生成对应数量的人物槽位。
- 每个人物槽可以独立绑定：
  - 人物形象图
  - 上身穿搭
  - 下身穿搭
  - 处理方式
  - 产品图
- 产品来源于顶部“产品图”板块。
- 人物 A/B、产品 A/B 使用内部标签隔离，提示词中禁止串脸、串服装、串产品、复制人物和身体融合。
- 提示词可读部分会移除 `[identity-lock]`、`[person-A]`、`[product-for-A+B]` 等机器标签。

### 2.5 姿势锁定重构的人物身份优先级

已经修复两个关键错误：

1. 前端曾把产品图和人物图的提示名称函数交叉使用。
2. 姿势锁定中的人物形象图曾被当作“人物+服装参考”，而不是纯人物身份参考。

现在的优先级为：

1. **人物形象图**：最高且唯一的人物身份来源，内部标记 `[identity-lock]`，权重 `high`。
2. **动作参考图**：只提供姿势、肢体角度、重心、手脚位置、头部朝向和动作。
3. **产品主图**：控制产品外观和结构。
4. **场景参考图或预设场景**：控制环境。

提示词明确要求：

- 完整丢弃动作图中原人物的身份、面孔、发型、肤色和身体特征。
- 强制替换为上传人物图中的人物。
- 禁止把动作图原人物、默认模特或其他人物槽的身份混入结果。
- 加强脸型、五官几何、眼鼻唇比例、发型、肤色和体型锁定。
- 加强面部表情自然度、眼神、嘴部肌肉、身体动力链、重心、关节和手部自然度。

姿势锁定已进一步简化人物位置参数：

- 单人不再需要位置输入，直接替换动作图中的人物。
- 双人按动作图“从左到右、由前到后”自动识别，按照人物 A、人物 B 的人物图上传/绑定顺序替换。
- 姿势锁定页面不再显示“原图位置”下拉框。

### 2.6 原图单品替换隔离与产品权重

这是最后一轮重点修复。此前用户的“原图单品替换”提示词中仍出现：

- 多人槽位绑定
- 上一轮人物图
- `[action]` 动作图
- `[composition]` 构图图
- `[scene]` 被当作普通场景参考，而不是不可变底图

这不是单纯缓存问题，而是实际编译和请求链路串用了旧数据。

现在已经完成：

- `product_only` 不再启用 `personBindings`、`multiPersonMode` 或人物参考图。
- 前端只激活一张原场景图和产品图；旧动作图、构图图、人物图保留在项目中，但在该工作流下不发送给模型。
- 服务端再次进行兜底过滤。即使浏览器或 LocalStorage 误提交旧人物槽、旧人物图、旧动作图和旧构图图，服务端也会剔除。
- 原场景图从普通 `[scene]` 提升为 `[base_image]`，权重强制为 `high`。
- `[base_image]` 被定义为不可变底图：目标产品区域之外的人员、脸、身体、姿势、穿搭遮挡、构图、场景、道具、机位、光影、阴影和像素都保持不变。
- 产品主图对目标产品区域拥有绝对优先级。
- 提示词强制先完整移除旧产品，再换成产品主图中的新产品。
- 明确禁止只改颜色、混合新旧产品、保留旧产品设计或直接返回未替换的原图。
- “视觉类型”中的通用替换规则也做了工作流覆盖，不再在 `product_only` 中出现“姿势、动作、机位、构图四项硬约束”的冲突文案。

### 2.7 场景职责隔离

- `[action]`：只能提供动作，不得提供场景、构图、人物身份、服装和光影。
- `[composition]`：只能提供机位、景别、裁切、主体占比、留白、透视和空间布局，不得提供背景内容和人物身份。
- `[scene]`：只能提供环境、地点元素、道具、光影、色彩和氛围。
- 没有上传 `[scene]` 时，预设场景是唯一环境来源，动作图和构图图禁止携带场景内容。
- `pose_rebuild` 只激活动作图和场景图，不会把以前上传的构图图带入。
- `product_only` 只激活 `[base_image]`。
- `multi_replace` 才会激活完整的多类别参考图。

### 2.9 创意模式人物动作合理性

根据最近生成图中出现的无支撑抬腿、重心不明、无意义手势、抓捏皮肤、布料受力不匹配和极限侧弯问题，已在创意模式的“质量底线”中加入人物动作合理性规则。

- 只在创意模式且画面人数大于 0 时启用。
- 创意无人模式不启用。
- 替换模式不启用，避免覆盖姿势锁定参考图；替换模式继续使用自己的表情与肢体自然度规则。
- 正向要求动作必须清晰可命名、符合场景，并具有可信支撑点、承重腿、重心和连续动力链。
- 手部接触必须具有真实触点、压力、遮挡和对应布料形变；非弹力展示时禁止无目的抓捏皮肤或拉扯腰头、裤脚。
- 未明确要求运动时优先自然站立、缓步、坐靠、整理衣物和轻度伸展，避免高抬腿、极限侧弯、悬空姿势和复杂瑜伽动作。
- 多人时要求人物动作独立但协调，禁止镜像复制和人物之间动作毫无关系。
- 中英文正向提示词和负向提示词均已同步。

## 6. 踩过的坑

### 6.1 产品图和人物图的提示名称函数曾交叉

`getPromptAssets` 中产品图曾调用人物命名函数，人物图曾调用产品命名函数。这会让模型误判素材职责，是姿势锁定不替换人物的重要根因之一。现在已经纠正。

### 6.2 人物身份图曾被误分类为服装参考

姿势锁定中的人物图必须是 `character_identity`，不能是 `character_garment_reference`。否则模型可能更关注穿搭关系而弱化身份替换。

### 6.3 “保持人物不变”与“替换人物身份”互相冲突

此前人物槽 `keep` 被编译为“完整保持该人物不变”，但姿势锁定实际要保持的是姿势和位置，人物身份必须替换。现在改为：保持姿势、位置、朝向和遮挡关系，身份强制替换为绑定人物图。

### 6.4 工作流 UI 隐藏不等于数据真正隔离

`product_only` 页面虽然不显示人物槽和动作/构图上传区，但旧数据仍存在于项目状态和 LocalStorage。必须在以下三层都过滤：

1. 前端提示词入参。
2. 前端二进制图片资产。
3. 服务端再次按工作流过滤。

只做其中一层会在旧项目、缓存状态或直接 API 请求中复发。

### 6.5 `[scene]` 不能同时表示“普通场景参考”和“不可变原图”

普通 `[scene]` 的职责是只提供环境，但原图单品替换需要整张图片除产品区域外都保持不变。这两个语义冲突。现在使用独立的 `[base_image]`。

### 6.6 提示词预览可能短暂显示上一轮内容

预览接口有约 260ms 防抖和异步请求。切换工作流时如果不清空旧预览，用户会误以为新工作流仍在使用旧提示词。现在切换时立即清空并显示加载状态。

### 6.7 多人位置参数不是越多越好

姿势锁定的核心是“动作图的人换成目标人物”，手动输入左/右/前景容易与图像实际关系冲突。当前改为自动顺序绑定；只有多要素精确替换仍保留位置参数。

### 6.8 Windows 控制台可能显示中文乱码

PowerShell 的 `Get-Content` 在部分输出中把 UTF-8 中文显示成乱码，但 `rg`、Git diff、TypeScript 构建和页面实际文本正常。不要只根据终端的乱码显示判断源文件已经损坏；需要结合 `git diff`、构建和页面验证。

### 6.9 不要破坏当前脏工作树

当前工作区有未提交修改，均属于本轮项目优化。不要使用 `git reset --hard`、`git checkout --` 等命令。继续修改前先查看 diff，并保留用户现有改动。

### 10.7 最新踩坑

#### 权重不能只显示在 UI 或文件名里

对模型而言，“权重：高”不等于强制执行。如果职责正文仍写“可以参考”，模型会忽略主体，只继承蓝白色调。权重必须同时进入中文、英文、逐图二进制说明和最终合规检查。

#### 场景提示词不能混入展示方式

“陈列台”看似环境描述，实际会强烈诱导单品静物构图，并与双人参考图冲突。场景只能定义环境；人物、动作、构图和产品展示方式必须由各自来源控制。

#### 0 人不等于删除参考图中的所有人

正确解释是禁止自主新增。如果写成绝对无人，会与有人的创意参考图或动作图直接冲突。

#### 只写“生成类似风格”太弱

模型可能只继承色调，却完全改变主体。高权重必须逐项写清人物数量、关系、动作、裁切、机位、道具、产品位置和构图骨架，并明确列出禁止退化的错误结果。

#### Windows 读取中文要指定 UTF-8

读取本文件或源码时优先使用 `Get-Content -Encoding utf8`。默认编码可能显示乱码，但不要因此判断文件损坏。
